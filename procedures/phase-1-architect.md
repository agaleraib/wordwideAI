# Phase 1 — Architect

> Purpose: Understand what you're building, for whom, at what quality bar, and write a spec an agent can execute against.
>
> Rule: Code written during Architect is a spike — temporary, not committed to main. If you find yourself committing real implementation here, the phase is over and you didn't close it properly.

## Entry gate (must be true to enter)

- [ ] `.harness-profile` exists (run `project-init` if not)
- [ ] Directory is a git repo with a clean working tree (or you know what your WIP is)
- [ ] A 1-4 sentence idea of what you want to build

## Required tools

| Tool | When | Why |
|---|---|---|
| `project-init` | If profile missing | Captures audience, stakes, data sensitivity, quality bar |
| `session-start` | Every session opening | Anchors today's goal in the profile context |
| `spec-planner` agent | Once, at phase start | Runs discovery (up to 3 rounds), writes dated spec to `docs/specs/YYYY-MM-DD-<topic>.md` |
| `micro` | Before each work block | Even during Architect — "research entity relationships" is a valid micro-goal |
| `park` | Whenever distractions surface | Keep spec work focused |

## Required artifacts (must exist to exit)

- [ ] **`docs/specs/YYYY-MM-DD-<topic>.md`** — generated by spec-planner, includes: overview, data model (if stateful), API surface (if backend/webapp), design principles (if UI), phased requirements with hard-threshold acceptance criteria, sprint contracts with Files/Depends-on/Verify, out-of-scope, open questions
- [ ] **`docs/architecture.md`** — single-page architecture overview (required for `project.type ∈ {backend, webapp, fullstack}`, optional for `script`)
- [ ] **`docs/plan.md`** — phase-1 task list pulled from the spec's sprint contracts (or per-workstream plans if `workstreams.mode = multi`)

## Exit gate (must be true to move to Code phase)

- [ ] Spec is approved by you (read it, make sure it matches your intent — do not skip this)
- [ ] Every phase-1 feature has **hard-threshold acceptance criteria** (not vague language)
- [ ] Every phase-1 task has Files / Depends-on / Verify filled in
- [ ] Out-of-scope section exists and lists at least 3 things you deliberately skipped
- [ ] Open Questions that block phase 1 are resolved; the rest are parked for later phases
- [ ] If `project.type ≠ script`: `docs/architecture.md` describes entities, data flow, and external services in one page
- [ ] Write `.harness-state/current_phase` = `Code`

## Common pitfalls

1. **Writing the spec in your head, not on disk** — if `docs/specs/` is empty, the phase is not complete, no matter how clear the idea feels.
2. **Acceptance criteria that aren't testable** — "users can search easily" is not a criterion. "`GET /search?q=foo` returns results containing 'foo' in the name field within 500ms" is.
3. **Skipping the data model** — even for "simple" projects, writing the entities down forces you to notice missing fields early.
4. **Letting discovery run forever** — spec-planner caps at 3 rounds. If you're on round 4, you're avoiding the commitment. Write the spec with the best guess and iterate.
5. **Starting Code phase before the architecture doc exists** — for anything bigger than a script, this guarantees later rework.
6. **No Out-of-Scope section** — the #1 cause of scope creep. If you can't list 3 things you're deliberately not building, you haven't thought hard enough.

## What this phase does NOT do

- Write real implementation code (spikes are OK, commits are not)
- Set up CI/CD (that's part of Deploy)
- Pick a deployment host (that's part of Deploy)
- Write tests (that's part of Code and Test)

## Transition to Code phase

When the exit gate is green:

1. Commit the spec + architecture + plan as a single atomic commit
2. Write `.harness-state/current_phase` = `Code`
3. Run `session-end` to close the day cleanly
4. Next session: `session-start` will read the new phase and orient you correctly
